Approach

CSS/HTML Styleguide

Basic syntax

  • Use soft-tabs with a two space indent. Spaces are the only way to guarantee code renders the same in each developer's environment.
  • If you write pure CSS or you're using CSS compatible SCSS syntax, always put spaces after : in property declarations and before { in rule declarations - this makes code easier to read. SASS syntax enforces you to place space after each semicolon.
  • End all declarations with a semi-colon (for CSS and SCSS). Despite that fact that the last declaration's is optional, it is better to put semi-colon at the end of each line.
  • Each CSS declaration should appear on its own line. Do not place whole ruleset on a single line, minifier will do that for you. Multiline blocks are easier to read and understand.
  • Comma-separated property values like multiple background or multiple box-shadows should include a space after each comma. But do not include spaces after commas in rgb(), hsl(), or deg() values.
  • Use lowercase for all hex values. #fff instead of #FFF. And use shorthand hex values where possible #fff instead of #ffffff. Remember that you can shorten hexes like #aabbcc to #abc
  • You can drop leading zero for opacity in color parameters (e.g. rgba(0,0,0,.4)) and transition durations (e.g. margin .5s linear)
  • User shorthand notations for margin, padding, font, background and border where available (margin: 5px 10px instead of margin: 5px 10px 5px 10px). And avoid specifying units for zero values (margin: 0; instead of margin: 0px;
  • Quote attribute values in selectors with double quotes (e.g. input[type="text"]). Even though they are optional in some cases, it is a good practice to use them for consistency.

Selectors

  • For styling purposes use classes only as CSS selectors. Using tag names as CSS selectors is a bad idea unless you want to specify initial styling of specific tag (e.g. typography rules for text and headings). Using tag names even if they are namespaced by a class name can lead to collisions and applying unwanted tag styles that were defined elsewhere in stylesheets.
    • 
        // Bad
        .navigation li a {
          ....
        }
      
    
        // Good
        .navigation .item {
          ....
        }
  • Using ids is also anti-pattern because it makes element not re-usable on that page. And you never know when you're going to need another identical component on the page (even multiple modals are possible). And also ids have grater specifity than classes so it will be harder to override them.
  • Always keep class names lowercase and use single dash between words. This look natural, it is easy to read and this is pretty common and nice CSS convention whilst camelCase is more JavaScript style and underscore_case is used in Ruby. A mix of couple of this styles in your stylesheets is a worst crime against consistency.
    • 
          // Bad
          .navBarWide .popup-close_XL
        
    
          // Good
          .navbar-wide .popup-close-xl
  • Keep classes as short as possible, but use only meaningful names. You don't need to be so verbose to define class like .top-navigation-bar-stripped instead of simple .navigation. But do not shorten it to .nav or even .n because it might not have sense for other developers working with you and will not have sense for you after some time.

  • Do not use presentational, positioning or purposeful class names. Presentation of various components changes over time as, but class names in html remain. And soon you'll find class .red with background: blue; in your code.

    As well as .products-list could be found in listing articles or tweets because these list are so similar in styles that you can reuse this class for all of them. It is better to focus on what this UI component is for, rather that what is listed in this list. And something like .entries-list will work for every listing in your app.

  • Distinguish your css classes and classes that are used in JavaScript. Use js- prefix to use in JavaScript, do not use 'style classes' there. And of course do not use js- classes in your stylesheets. This will help you not to break functionality if you decide to rename or remove some of CSS classes.

HTML Simplicity

  • Keep your HTML structure as simple as possible. Use HTML tag for content markup in first place without thinking about presentational layers that may be needed in the future. Minimize number of wrappers that are needed only for styling purposes. Consider using pseudo elements ::after and ::before to move all decorative code from HTML to CSS.
  • Do not overload HTML tags with long chains of CSS classes. If you are following component approach there shouldn't be such case in your code, when you need to apply more than 3 classes to single HTML tag. All components can be defined by applying 2 classes - component name class and component modifier (if default component needs modification).
